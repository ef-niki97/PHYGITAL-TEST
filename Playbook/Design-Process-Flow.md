# Design Process Flow — гайд по рабочему процессу в дизайне

Источник: [Figma/FigJam-доска "Design Process Flow"](https://www.figma.com/board/URGM77tQBU9qyOd64ZAABV/Design-Process-Flow?node-id=0-1&t=exsZGz0S1EJNjU0Z-1), страница "General Flow". Этот файл — текстовое переложение схемы для плейбука.

> В самой схеме есть авторская пометка: *"на стикерах оставила пока пояснения, они усложняют восприятие схемы, сейчас я бы информацию с них перенесла в плейбук"*. Этот документ — как раз перенос этих пояснений сюда.

На доске также есть вторая страница **"Responsibility"** (зоны ответственности), которая не открылась в режиме просмотра без авторизации в Figma — стоит добавить её содержимое сюда отдельно, когда будет доступ.

## Общая картина: 10 этапов

Процесс идёт по 10 шагам от постановки задачи до финализации. Каждый шаг привязан к автоматизациям в ClickUp/Figma/Slack, которые двигают задачу по статусам без ручной рутины.

1. **Постановка задачи**
2. **Начало работы: сбор референсов**
3. **Обсуждение референсов и первичных концептов: выбор направления**
4. **Создание вариантов для реализации задачи**
5. **Выбор финального решения**
6. **Перевод макетов на компоненты**
7. **Финальное ревью**
8. **Создание задачи на разработку**
9. **Проверка реализации**
10. **Финализация**

Шаги 2–5 визуально объединены на схеме (общая "ромбовидная" зона) — это дискавери/концептинг-фаза, где нормально возвращаться на шаг назад, если решение не найдено.

## Разбор по шагам

### 1. Постановка задачи
Задача оформляется в ClickUp: заводится карточка с описанием задачи (**ClickUp → "Описание задачи"**).

### 2. Начало работы: сбор референсов
- В Figma создаётся рабочий файл на основе шаблона (**Figma → "Создать шаблон на базе текущего (ссылка)"**).
- Задача в ClickUp двигается в статус **in progress**.
- Параллельно собираются референсы в отдельном Figma-файле (**"Собрать референсы"**).

**Правила создания файлов:** файл создаётся в папке по проекту/направлению, в названии обязательно указывается задача, чтобы файл можно было сопоставить с ClickUp-карточкой.

**Правила сбора референсов:** сохранять оригинальное название источника, добавлять комментарий — откуда референс и чем важен, располагать референсы так, чтобы их было легко найти повторно.

### 3. Обсуждение референсов и первичных концептов: выбор направления
Команда/лид обсуждает собранные референсы и черновые концепты, выбирается направление для проработки вариантов.

### 4. Создание вариантов для реализации задачи
Прорабатываются варианты решения. Здесь же, если нужно быстро проверить интерактивность идеи, собирается кликабельный прототип через AI-инструменты — **Claude, Cursor, Lovable** ("Собрать прототип").

**Правила сбора прототипа:** прототип должен быть интерактивным и соблюдать дизайн-систему.

### 5. Выбор финального решения
- Финальные варианты собираются в Figma (**"Собрать варианты"**).
- К ревью макета подключаются разработчики — для оценки реалистичности/фичасибилити реализации (**"Подключить разработчиков к ревью макета для оценки реалистичности"**).
- Если однозначного победителя нет — включается **тестовый цикл**:
  - **5.1 Выбор нескольких решений на тесты**
  - **5.2 Запуск исследования и результаты**
  - результаты исследования возвращают процесс к выбору финального решения.

### 6. Перевод макетов на компоненты
Финальный макет собирается на компонентах дизайн-системы (подготовка к разработке).

### 7. Финальное ревью
Макет проходит ревью у лида или ответственного дизайнера — по итогам принимается решение о готовности к передаче в разработку. Задача в ClickUp двигается в статус **review**.

### 8. Создание задачи на разработку
Для разработчика создаётся новая задача в ClickUp с тегом **design**, которая связывается с исходной дизайн-задачей:
- **ClickUp:** задача дизайнера переводится в статус **to prod**
- **ClickUp:** создаётся новая задача на разработку с тегом **design**
- **ClickUp:** задача разработки прикрепляется в комментарии к исходной задаче

### 9. Проверка реализации
- **ClickUp:** задача разработки взята в работу и переведена в статус **QA**
- Дизайнер проверяет реализацию на деве — соответствует ли макету
- Развилка:
  - **"Все ок"** → переход к шагу 10
  - **"Доработка нужна"** → задача возвращается в разработку (обратно к шагу 8/9)

### 10. Финализация
Финальный перевод задачи в завершающий статус в ClickUp.

## Процесс review (фидбэк-цикл)

Отдельный под-процесс, который используется при передаче наработок на ревью (в первую очередь вокруг шагов 5–7):

1. **Поделиться наработками с командой** — ссылкой в Figma, ссылкой на прототип, картинками.
2. **Дизайн ревью** — разбирается тремя параллельными способами:
   - **Комментарии в файлах от лида** (в Figma)
   - **Обсуждение с лидом**
   - **Ревью ИИ агента** — отдельный AI-агент проверяет работу как ещё один ревьюер
3. Обсуждение дублируется в тред в Slack: **#design_product / #design_graphic**.

## Текущие инициативы (снэпшот с той же доски, "3 больших процесса")

Это не шаги процесса, а список параллельных крупных инициатив на момент создания схемы — полезно как контекст, но стоит сверять с актуальным статусом:

1. **Figma ⇒ code** (Никита, Даня, Андрей) — компоненты берутся из дизайна в Figma так, чтобы код полностью соответствовал Figma.
2. **Redesign / New UI Kit "Phygital 2.0"** — выбор библиотеки для нового UI Kit, кликабельные прототипы для тестов, ТЗ, проблемы, тесты концептов, гипотезы.
3. **Переход на React / новый движок** — выбор библиотеки UI Kit под базовые компоненты.
